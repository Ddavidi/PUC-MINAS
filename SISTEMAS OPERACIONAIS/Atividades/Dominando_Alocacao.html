Dominando Alocação de Arquivos

Como o SO decide onde gravar os dados no disco físico (Visão de Tanenbaum)

1. Alocação Contígua

O arquivo inteiro é gravado em blocos sequenciais e vizinhos no disco. Se precisa de 50 blocos, eles têm que estar colados um no outro.

✅ Vantagens

  • Desempenho Extremo: Leitura mais rápida de todas (1 único seek e lê direto).
  • Implementação: Muito simples. O diretório só anota o bloco inicial e o tamanho.

❌ Desvantagens

  • Fragmentação Externa: O disco vira um queijo suíço. Buracos vazios se espalham, e arquivos novos não cabem.
  • Crescimento Impossível: Se o arquivo crescer e o bloco vizinho estiver ocupado, o SO tem que copiar o arquivo inteiro para outra área.

2. Alocação Encadeada (Lista Ligada)

O arquivo é espalhado em blocos isolados. O truque: o primeiro bloco guarda os dados e um ponteiro (link) físico apontando para o endereço do próximo bloco no disco.

✅ Vantagens

  • Zero Fragmentação Externa: Qualquer buraco livre serve. Acabou o queijo suíço.
  • Crescimento Fácil: O arquivo cresce indefinidamente achando blocos soltos.

❌ Desvantagens

  • Acesso Randômico Destruído: Para ir ao bloco 50, o SO é obrigado a ir no disco e ler fisicamente o bloco 1, 2, 3... até chegar no 50. Muito lento!
  • Overhead Interno: O ponteiro rouba espaço dos dados, quebrando a potência de 2 do bloco (ex: de 512 sobram 508 bytes).

3. Alocação Encadeada com Tabela em Memória (FAT)

Evolução do anterior. Arranca-se os ponteiros de dentro dos blocos no disco, e coloca-se todos juntos numa tabela (File Allocation Table), carregada inteira na Memória RAM.

✅ Vantagens

  • Salvação do Acesso Randômico: O SO percorre a corrente de ponteiros na RAM (quase instantâneo) e vai no disco apenas 1 vez, direto no alvo.
  • Blocos Íntegros: O bloco volta a ser 100% livre para dados de usuário.

❌ Desvantagens

  • Gasto de RAM Alto: A tabela precisa mapear o disco inteiro na RAM o tempo todo. Discos muito grandes esgotam a memória facilmente.

4. Alocação Indexada (i-nodes do UNIX)

A solução definitiva: cria-se um bloco especial no disco (o i-node ou bloco de índice) para CADA arquivo. Esse bloco guarda o endereço de todos os blocos de dados do arquivo.

✅ Vantagens

  • Economia de RAM: A tabela não é global. O SO só carrega o índice (i-node) para a RAM se o arquivo for aberto pelo usuário.
  • Acesso Randômico Direto: Lê-se a linha do índice e o ponteiro já dá o endereço final do disco.

❌ Desvantagens

  • Fragmentação Interna / Overhead: Um arquivo minúsculo gasta 2 blocos (1 pro dado e 1 inteiro só pro índice).
  • Arquivos Gigantes: Requer índices multiníveis (blocos de índices apontando para mais índices).

📌 Resumo Matador para Fixar

Alocação Contígua: Perfeita para ler rápido. Inviável se o arquivo precisa crescer. Sofre com fragmentação externa.
Alocação Encadeada (Disco): Excelente para crescer. Péssima se você quiser dar pulos randômicos no arquivo.
Encadeada com Tabela (FAT): Resolve o acesso randômico botando os ponteiros na RAM, mas consome a RAM inteira.
Indexada (i-nodes): Resolve todos os problemas anteriores, gastando RAM só com os arquivos que estiverem abertos (mas gasta espaço extra no disco com índices).
